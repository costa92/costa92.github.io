vLLM vs Ollama vs llama.cpp：DevOps 怎么选

上周又有同事在群里问：自建大模型服务，到底是上 vLLM 还是 Ollama？要不要试试 llama.cpp？

这个问题我自己踩过坑——一开始拿 Ollama 怼生产，并发一上来延迟直接飙红；后来全栈换 vLLM，结果 CI 机器上根本没 GPU，编译都跑不起来。折腾完我才明白：这三个东西压根不是竞品，是一条流水线上的三个工位。

很多选型文章喜欢搞个评分表，最后给个”综合最强”的冠军。我觉得这思路从一开始就错了。它们各自优化的目标根本不在一个维度上：vLLM 优化的是生产吞吐（一个服务能扛 50+ 并发），Ollama 优化的是开发者上手体验（一条 ollama run 就开干），llama.cpp 优化的是 CPU 效率和可移植性（4.5GB 内存跑 7B）。拿吞吐去比 Ollama、拿易用性去比 vLLM，就像拿货车跟跑车比谁省油——问题本身就问歪了。

先把结论摆这儿，不绕弯子：本地开发拿 Ollama，CI 和边缘设备用 llama.cpp，生产高并发上 vLLM。模型文件三者基本通用，换运行时不是重写架构，更像换个发动机。

下面我把三个发动机一个个拆开看，顺便纠正两个流传很广的错误说法。

vLLM：为高并发而生的吞吐怪兽

如果你的场景是”一个服务，几十上百号人同时调”，那 vLLM 几乎没有对手。

它快的核心是 PagedAttention。官方文档原话是 “Efficient management of attention key and value memory with PagedAttention”——说白了，传统推理给每个请求的 KV 缓存预留一整块连续显存，请求一多显存就碎成渣、白白浪费。PagedAttention 把 KV 缓存当成操作系统的”虚拟内存页”来管，按需分配，碎片几乎归零。

显存省下来了，就能塞更多请求一起算，这就是第二招 连续批处理（continuous batching）。官方功能列表里写的是 “Continuous batching of incoming requests, chunked prefill, prefix caching”。

这里值得多说两句，因为它是 vLLM 吞吐高的真正原因。传统的”静态批处理”是一批请求必须凑齐、一起进、一起出——其中一个请求生成的 token 特别长，整批都得等它，GPU 大部分时间在干等。连续批处理则是”谁生成完谁就走，空出来的位置立刻补新请求进来”，GPU 利用率几乎拉满。

配套的还有两个细节：prefix caching 会把相同的前缀（比如一段几百 token 的 system prompt）算一次就缓存住，后面命中的请求直接复用；chunked prefill 把长输入切块处理，避免一个超长 prompt 把整批的解码都卡死。这三招叠在一起，才是那个夸张吞吐数字的来源——后面 docker run 起服务时这些默认就开着，你不用调。

这里要纠正一个常见说法：很多文章引用 vLLM “吞吐提升 23 倍”。这个 23x 数字其实来自 Anyscale 的一篇外部博客（讲连续批处理的），不是 vLLM 文档自己的实测，引用时最好把出处带上。

部署上 vLLM 走 Docker 最省事，起来就是一个 OpenAI 兼容的 API：

docker run --runtime nvidia --gpus all \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  -p 8000:8000 \
  vllm/vllm-openai:latest \
  --model mistralai/Mistral-7B-Instruct-v0.2

起来之后，客户端代码跟调 OpenAI 没区别，迁移成本几乎为零：

curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "mistralai/Mistral-7B-Instruct-v0.2",
    "messages": [{"role": "user", "content": "你好"}]
  }'

官方文档还提到，除了 OpenAI 接口，它同时支持 “Anthropic Messages API and gRPC support”，接入选择很宽。

性能数字我引一下源文章的实测（注意：这是第三方博客数据，我没有独立复现）：A10G 显卡跑 Mistral 7B、batch 16，吞吐约 1800 tokens/秒，p50 延迟 820ms、p99 1450ms，50+ 并发不掉速。数量级可信，具体数字以你自己的硬件为准。

最后纠正第二个误区——vLLM 不是只能用 NVIDIA。这是流传最广的错误。官方明确写支持 “NVIDIA GPUs, AMD GPUs, and x86/ARM/PowerPC CPUs”，还有 TPU、Apple Silicon 等一堆插件。当然生产主流仍是 N 卡，但”必须 N 卡”这话不成立。

运维上有几个坑我提前说一下。第一，显存要算清楚——模型权重之外，KV 缓存会随并发和上下文长度涨，--max-model-len 设太大很容易 OOM，上线前务必按峰值并发压一遍。第二，单卡放不下的大模型用张量并行 --tensor-parallel-size 2 拆到多卡，但跨卡通信会吃掉一部分收益，不是卡越多越线性。第三，它是常驻显存的重型服务，别指望像 Ollama 那样用完即走——这是生产服务该有的样子，但也意味着你得为它专门留机器。

我的判断很直接：只要是面向多人的线上推理，我一律上 vLLM，不会用别的。 它上手确实比另外两个陡，但生产环境要的就是吞吐和稳定性，这块它做到了。

反过来，单人用、低流量的场景别上 vLLM，纯属杀鸡用牛刀。 它的所有设计——分页显存、连续批处理——都是为”很多请求挤在一起”准备的。你就一个人偶尔调一下，这些机制全是空转，还得伺候 GPU 和一堆启动参数，性价比极低。这种场景 Ollama 一条命令的事。

llama.cpp：没有 GPU 也能把模型扛起来

vLLM 再强，遇到”机器上没 GPU”就歇菜。这时候 llama.cpp 是我唯一的选择。

它的定位官方一句话：”LLM inference in C/C++”。纯 C/C++ 写的，没有那堆 Python 依赖，CPU 上就能跑。这让它特别适合两个场景：CI 流水线里跑模型冒烟测试，以及边缘设备/Mac 上做本地推理。

能在 CPU 上跑大模型，靠的是 GGUF 量化。官方支持从 “1.5-bit, 2-bit, 3-bit, 4-bit, 5-bit, 6-bit, and 8-bit” 的整数量化——把原本 FP16 的权重压成低比特整数，体积和内存占用断崖式下降。源文章给的例子是 Mistral 7B 的 Q4_K_M 格式只占 4.5GB 内存（这个具体档位官方页没单列，属 GGUF 常见量化档）。

量化档位怎么选，是用 llama.cpp 绕不开的决策。规律很简单：比特越低，体积越小、速度越快，但模型质量损失越大。 我的经验是 Q4_K_M 是甜点档——7B 模型压到 4-5GB，质量损失肉眼基本看不出，普通笔记本就能跑。要更高质量上 Q5、Q6，机器内存够就 Q8（接近原始精度）；2-bit、3-bit 那种极限压缩除非内存实在紧张，否则别碰，模型会开始说胡话。

如果机器上有一块不大的显卡，还能用 -ngl 参数把一部分层 offload 到 GPU，剩下的留在 CPU，混合着跑：

# 把 20 层放到 GPU，其余在 CPU，适合显存不够装下整个模型的情况
./build/bin/llama-server -m model.gguf -ngl 20

编译和起服务都很直接：

git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp
cmake -B build && cmake --build build --config Release

llama.cpp 自带一个 OpenAI 兼容的 HTTP 服务，官方描述是 “A lightweight, OpenAI API compatible, HTTP server for serving LLMs”：

# 直接从 HuggingFace 拉 GGUF 模型起服务
./build/bin/llama-server -hf ggml-org/gemma-3-1b-it-GGUF

想跑一次性推理不开服务，用 CLI 就够：

./build/bin/llama-cli -m my_model.gguf -p "解释一下什么是量化"

性能方面，源文章说现代 CPU 上 15-25 tokens/秒，Apple Silicon（M2 Pro，走 Metal）能到 50 tokens/秒。我得提醒一句：llama.cpp 官方 README 里有 llama-bench 的示例跑分，但没有针对”现代 CPU / Apple Silicon 通用速度”的官方实测数字，文中这些 15-25 / 50 tokens/秒 都来自第三方，自己的机器请用 llama-bench 实测。不过 Apple Silicon 的优化是官方实锤的——文档里写 “Apple silicon is a first-class citizen - optimized via ARM NEON, Accelerate and Metal frameworks”。在 Mac 上跑本地模型，它确实是第一梯队。

放进 CI 的价值我特别想强调。模型上线前最怕”换了个量化版本，结果输出全乱了”这种事。在 CI 里用 llama.cpp 加载目标 GGUF、跑一条固定 prompt、断言输出非空且格式正确，这一步不需要 GPU、几秒钟跑完，能挡住一大半”模型文件本身坏了”的低级事故。我现在的项目都把这步当成模型版本的冒烟测试。

我的判断：llama.cpp 不追求并发吞吐，它追求的是”在任何破机器上都能把模型跑起来”。CI 没 GPU？边缘盒子只有 ARM？数据敏感不能出本机？这活儿基本只有它和 Ollama 干得了，而要塞进自动化流水线、不带多余依赖，一个 llama-cli -m model.gguf 就能跑的它更纯粹。

Ollama：本地开发，五分钟跑起来

如果说 vLLM 是给运维的、llama.cpp 是给嵌入式的，那 Ollama 就是给开发者图省事的。

我自己日常本地调模型，第一反应永远是 Ollama，因为它把所有麻烦事都藏起来了。一条命令拉模型、起服务、进对话：

ollama run gemma3

模型从 ollama.com/library 拉，名字一报就行。它现在的明星阵容（官方 About 原话）是 “Kimi-K2.6, GLM-5.1, MiniMax, DeepSeek, gpt-oss, Qwen, Gemma and other models”，主流开源模型基本都在。

更关键的是它内置 REST API，官方原话 “Ollama has a REST API for running and managing models”，而且带 OpenAI 兼容端点（路径就在 /v1）。本地写的代码，将来切到 vLLM 接口不用动——直接用这个兼容端点，跟前面 vLLM 的请求体长得一模一样：

curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma3",
    "messages": [{"role": "user", "content": "写一个快速排序"}]
  }'

想自定义模型行为，用 Modelfile（类似 Dockerfile 的思路）：

FROM gemma3
PARAMETER temperature 0.7
SYSTEM "你是一个简洁的代码助手"

顺带说个很多人不知道的点：Ollama 底层其实就是封装了 llama.cpp。所以它跑的也是 GGUF 量化模型，能在 CPU 上跑、Apple Silicon 上快，都是继承自 llama.cpp 的能力。Ollama 在它之上加了模型管理、自动显存调度、热加载这些”开发者体验糖”——你只管 ollama run，模型拉取、量化档选择、显存分配它全替你办了。

它还有几个让本地开发特别舒服的小设计：模型默认会在内存里驻留一段时间（keep_alive），短时间内反复调不用重新加载；多个模型可以并存，按需换出。这些对本地反复试验太友好了。

还有一点对很多团队是刚需——全程本地、数据不出门。代码、内部文档这些敏感东西喂给云端 API 总让人心里没底，用 Ollama 在自己机器上跑，数据一个字节都不外传。对合规要求严的场景，这个特性比性能更重要。175k stars 的社区也意味着新模型出来基本第二天就能 ollama pull 到，跟进速度极快。

至于它的短板，前面说过了，这里再钉一下：它不是为并发设计的。我那次线上事故的根因就是把”单人爽”误当成”能扛量”。记住它的角色——开发期的加速器，到此为止。

截至写稿，Ollama 最新版是 v0.30.10（2026-06-17 发布），仓库 175k stars——社区热度它是三者里最高的。

但别被热度骗了，我不推荐拿 Ollama 直接上生产。它优化的是”上手快、单人爽”，不是高并发吞吐。我那次拿它扛线上流量、并发一冲延迟就崩，就是没认清它的定位。它是开发期的瑞士军刀，不是生产期的发动机。

一张表看清三者差异

口说无凭，把关键维度拍成表（其中性能数字来自源文章第三方实测，仅供量级参考）：

维度vLLMllama.cppOllama
定位生产高并发CPU/边缘/CI本地开发
硬件N卡/A卡/CPU 都支持CPU 优先，Apple Silicon 一流GPU + CPU
核心技术PagedAttention + 连续批处理GGUF 量化封装 + 一键体验
APIOpenAI / Anthropic / gRPCOpenAI 兼容REST + OpenAI 兼容
上手难度高（要懂 GPU/K8s）中（要编译）极低（一条命令）
并发能力极强（50+ 不掉速*）弱弱
GitHub stars——118k175k

*并发数字来自源文章实测，未独立复现。

表里我特意把 API 那行标出来，因为这是三者能组成流水线的关键——它们都提供 OpenAI 兼容接口。这意味着你的业务代码只认一套接口，改个 OPENAI_BASE_URL 就换了底层运行时，代码一行不动。对 DevOps 来说这是天大的好事：不用为换运行时改业务逻辑，也不会被某一家锁死。

我自己有个真实的演进路线可以参考。最早做一个内部文档问答，本地用 Ollama 把流程跑通，三天就有了能演示的 demo。要进 CI 做回归测试时，CI 机器没 GPU，换 llama.cpp 加载同一个模型的 GGUF 版本，跑个最小推理验证模型没崩。最后上线给全公司用，并发压力上来了，换 vLLM 部署到 K8s。整个过程模型选型没变，业务代码因为都是 OpenAI 接口几乎没动，每一步只是换了个运行时——这就是把它们当流水线用的实际收益。

三者最妙的地方是模型可以平滑流转。你在 Ollama 上选定的模型，导出成 GGUF 就能喂给 llama.cpp，HuggingFace 上的原始权重又能直接喂给 vLLM。换运行时，配置改几行就行，比如生产的 K8s 里 vLLM 作为主服务、Ollama 作为开发 sidecar：

# 开发环境：Ollama 作为轻量 sidecar
containers:
  - name: ollama
    image: ollama/ollama:latest
    ports:
      - containerPort: 11434

而 CI 里则用 llama.cpp 做一次无 GPU 的推理冒烟测试：

# CI 步骤：无需 GPU，验证模型能正常出 token
./build/bin/llama-cli -m model.gguf -p "ping" -n 16 || exit 1

最后：我会怎么选

不绕了，直接给场景化推荐：

场景我的选择原因
本地写代码、调 promptOllama一条命令起步，省心
CI 流水线、边缘设备、无 GPUllama.cpp纯 C/C++，哪都能跑
团队共享、生产高并发vLLMPagedAttention 吞吐碾压
Mac 上跑本地模型llama.cppApple Silicon 一流优化

判断逻辑其实可以压成三行：

有没有高并发生产需求？ 有 → vLLM
机器上有 GPU 吗？      没有 → llama.cpp
就想本地快速试试？     是  → Ollama

再补一个很多人忽略的成本维度。vLLM 要常驻 GPU，机器成本最高，但摊到每个请求上反而最便宜——因为它把 GPU 榨得最干（高并发时 nvidia-smi 看显存利用率基本贴满），单 token 成本最低。Ollama 和 llama.cpp 单机轻量、几乎零运维成本，但它们不为吞吐设计，硬要扛大流量等于用算力换便利，反而不划算。所以”省钱”在不同规模下答案是反的：小流量用轻量方案省钱，大流量用 vLLM 才省钱。

还有人会问：能不能只用一个、全场景通吃？技术上当然行，但我不建议。拿 vLLM 在本地开发，每次改代码都要重启一个吃满显存的重服务，体验很差；拿 Ollama 上生产，并发一冲就现原形。每个工具在它不擅长的场景都会让你难受——这正是”三个都装、按阶段用”比”All in 一个”更省心的原因。

迁移成本低到什么程度？同一个模型，开发期在 Ollama，上线时换 vLLM，客户端代码因为都是 OpenAI 兼容接口，基本一行不用改：

# 开发：指向本地 Ollama
export OPENAI_BASE_URL=http://localhost:11434/v1
# 生产：指向 vLLM，代码不变
export OPENAI_BASE_URL=http://vllm-svc:8000/v1

最后说句公道话，这三个工具也有它们都不管的事，别指望换个运行时就万事大吉。模型本身的质量它们一个都不负责——选错了基座模型，跑得再快也是快速地输出废话。鉴权、限流（超额了得自己返 429）、可观测性、灰度这些生产必备的东西，三者也都只给你一个裸接口，得你自己在外面套一层。运行时只解决”怎么把模型跑起来、跑多快”，”跑得对不对、稳不稳”还是你的活。

所以别再纠结”三选一”了。我的建议是三个都装上，按阶段用——这才是把它们当成一个家族的正确姿势。

把这篇转给那个还在纠结自建大模型用哪个运行时的同事，能省他不少试错时间。
