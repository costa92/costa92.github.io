CodeGraph 实战：给 AI agent 装上代码地图，tool calls 砍掉一半

让 Claude Code 改一个稍微大点的项目，你大概都见过这个画面：它先 ls 一下、再 grep 几个关键词、然后挨个 read_file，把调用链在脑子里拼出来——真正动手改之前，已经烧掉十几个 tool call。

代码它早晚能找到，问题是这趟”探路”既慢又费 token。我自己在一个几千文件的仓库里盯着 agent 干活，经常一半时间它都在翻文件而不是写代码。

CodeGraph 想干掉的就是这段浪费：把任意一个文件夹预先解析成一张代码知识图谱，通过 MCP 喂给 agent。下面这篇我按它的官方最新版本走一遍——提醒一句，网上不少教程里的 codegraph init -i 之类命令已经过时了，以官方资料为准。

它解决的是 agent “找代码” 的浪费

先把问题说清楚。没有索引的时候，agent 理解一段陌生代码，靠的是内置工具一遍遍地搜：

# agent 回答"请求是怎么到数据库的"——典型的探路过程
grep -r "Database" src/
ls src/services/
# 然后挨个 read 命中的文件，自己在上下文里重建调用链……

CodeGraph 的思路是把这件事提前做掉：它一次性扫完整个仓库，记下每个符号（函数、类、方法）、每条调用边、每个依赖关系，存成一张图。agent 要查的时候问一句，直接拿回相关源码、调用路径、还有改动的”波及范围”（blast radius）——不用再一个文件一个文件地翻。

官方在 7 个真实开源项目上做了对比（Claude Code headless，每组取 4 次中位数），结论挺直接：tool calls 中位数少 58%、回答快 22%、文件读取从一堆降到接近 0。

项目语言 · 规模tool calls文件读取token
VS CodeTS · ~1万文件少 81%0 vs 9少 64%
DjangoPython · ~3千少 77%0 vs 9少 60%
TokioRust · ~790少 57%0 vs 8少 38%
AlamofireSwift · ~110少 58%0 vs 9少 64%

官方自己也很诚实地标了一句：省 token、省钱是”规模相关”的——小仓库上每次查询省的那点钱又小又飘，得是大到 Google/Microsoft 那种单体仓库、再乘上整个团队每天的用量，才会攒成账单上的一笔。所以小项目用它，图的是快和准，不是省钱。这点比那些张口就是”省一半成本”的宣传实在得多。

你可能会担心一个问题：图是预建的，那我刚改完文件、图还没更新的那一小会儿，agent 会不会拿到旧答案？CodeGraph 在这里做得挺细——改动落盘后监听器很快触发（默认防抖 2 秒），而在这段空窗里，如果 agent 查到一个还没同步的文件，返回结果会顶一个 ⚠️ 提示，明确告诉它”这个文件刚改过，直接 Read 一下”。实测里 Claude Code 真的会照做，先去读实时内容再回答。不会出现那种”索引过期了还一本正经给你错答案”的情况。

还有个容易被忽略的场景：你在终端 git pull 了一把、或者用别的编辑器改了几个文件，这期间 MCP server 可能根本没在跑。CodeGraph 在 server 每次重连时会先做一轮 (大小, mtime) + 内容哈希的核对，把这些”背着它发生”的改动在回答第一个查询前补进来。所以不管改动是谁干的、什么时候干的，下次一问都是新的。

三步把 CodeGraph 接上 Claude Code / Cursor

接入分得很清楚：装 CLI → 连 agent → 给项目建索引，三件事，别混在一起。

第一步，装 CLI。它自带运行时，本机没有 Node 也能跑：

# macOS / Linux —— 一行装好，自动挑对应你系统的构建
curl -fsSL https://raw.githubusercontent.com/colbymchenry/codegraph/main/install.sh | sh

# 已经有 Node？用 npm 也行（任意版本都可以）
npm i -g @colbymchenry/codegraph

装完记得新开一个终端——安装脚本把 codegraph 放进了 PATH，但不会改当前这个 shell，不换终端下一步会找不到命令。

第二步，把 CodeGraph 连到你用的 agent 上。这一步才是”接线”，只装 CLI 是不够的：

codegraph install

它会自动探测你机器上装了哪些 agent——Claude Code、Cursor、Codex CLI、opencode、Gemini CLI、Antigravity、Kiro、Hermes Agent 都支持——然后把 MCP server 配置写进各自的配置文件。注意这步只接线、不建索引，全局跑一次就覆盖所有项目。

第三步，进到具体项目里建图。这一步才真正扫你的代码：

cd your-project
codegraph init

codegraph init 一条命令搞定两件事：建出本地 .codegraph/ 目录，并把整张图建好。建完之后自动同步默认就开着——文件改了它用系统原生事件（FSEvents / inotify）盯着，防抖一下增量更新，索引永远是新的，没有什么 sync 要你手动跑。把 agent 重启一下让 MCP 加载，就能用了。

如果是配进 CI 或者脚本里，install 支持完全非交互的写法，指定目标和范围就行：

# 自动探测 agent、装到全局，全程不问
codegraph install --yes

# 只接指定的几个，且只对当前项目生效
codegraph install --target=cursor,claude --location=local

想升级直接 codegraph upgrade，它会自己认出你当初是怎么装的（bundle / npm / npx）就地更新；哪天不想用了，codegraph uninstall 会把它写进各个 agent 的那些配置干净地撤掉，项目里的 .codegraph/ 索引留着不动。想连某个项目的索引一起清掉，进那个项目跑 codegraph uninit 就行——两件事分开，删 agent 配置和删项目图谱互不牵连。

一次查询，拿到答案而不是一堆文件

接好之后，agent 在回答结构性问题时就会自动走 CodeGraph，不用你操心。它对外其实只暴露一个 MCP 工具——codegraph_explore。

这是我觉得设计上很克制的一点。官方实测发现，给 agent 一堆细分工具（查调用者、查被调用、查依赖……）反而容易选错，不如一个够强的工具好用。所以 explore 一次调用就把相关符号的源码（按文件分组）、它们之间的调用路径（包括 grep 跟不动的动态分发）、加上波及范围一起返回。

你也可以直接在命令行里手动试，输出和 agent 拿到的一模一样：

# 一句话问清楚某个流程
codegraph explore "请求是怎么经过中间件链的"

# 查一个符号会牵动谁——改之前先看波及范围
codegraph impact UserService

# 全文搜符号，FTS5 驱动，瞬出
codegraph query "login" --kind function

单体仓库或者多仓库场景也照顾到了：只要某个目录建过 .codegraph/ 索引，查询时传一个 projectPath 就能命中它——一个会话里既查主服务、又查另一个子服务都行。碰到没建过索引的路径，它不会报错，而是回一句”这块用你的内置工具吧”，建不建索引始终是你自己决定。

如果你想手动接 Claude Code，配置也就这么一段——往 ~/.claude.json 里加：

{
  "mcpServers": {
    "codegraph": {
      "type": "stdio",
      "command": "codegraph",
      "args": ["serve", "--mcp"]
    }
  }
}

“只有一个工具”其实是默认姿态，不是全部。CodeGraph 另外还藏了 7 个更细的工具——codegraph_node、codegraph_search、codegraph_callers、codegraph_callees、codegraph_impact、codegraph_files、codegraph_status。它们默认不挂到 MCP 上，因为这些信息 explore 的返回里基本都带了。真要单独打开，用一个环境变量点名即可：

# 按需把额外的工具挂回 MCP surface
export CODEGRAPH_MCP_TOOLS=explore,node,search,callers

手动接的话，顺手在 ~/.claude/settings.json 里给它的工具开个 auto-allow，省得每次调用都弹确认（用 codegraph install 接 Claude Code 时这步会自动做掉）：

{
  "permissions": {
    "allow": ["mcp__codegraph__*"]
  }
}

还有个我觉得很实用的命令是 codegraph affected：顺着 import 依赖往上追，找出某次改动会影响哪些测试文件。配进 CI 或者 git hook，改完只跑受影响的那部分测试，不用全量：

# 从 git diff 喂进去，只跑被波及的测试
git diff --name-only HEAD | codegraph affected --stdin --quiet

真正让我觉得”这东西比 grep 强一个量级”的，是它能跨语言边界把调用链接上。纯靠 tree-sitter 静态解析，到语言边界就断了——Swift 调一个被自动桥接的 ObjC 方法、JS 通过 React Native 桥调原生模块、Expo Modules 的那套 DSL，grep 全都跟不动。CodeGraph 把这些 hop 显式补进图里，explore 出来的调用路径和波及范围能直接穿过去。它还认得 17 种 Web 框架的路由写法：Django 的 path()、FastAPI 的 @app.get()、Express 的 router.post()……查一个 controller 的调用者，会顺带把绑它的 URL 模式一起带出来。这种”动态分发 + 路由绑定”的关系，正是人肉 grep 最容易漏的部分。

原理：tree-sitter 解析 + SQLite 图 + 自动同步

底层其实不神秘，四步：tree-sitter 按语法解析出 AST，抽取节点（函数/类/方法）和边（调用/导入/继承）；全部塞进本地一个 SQLite 库（.codegraph/codegraph.db，带 FTS5 全文搜索）；再做一轮引用解析，把调用对上定义、导入对上源文件；最后文件监听做增量同步。

# 想确认索引状态，随时看
codegraph status

# 关掉匿名遥测（它默认会收集用了哪些命令/语言，但绝不碰你的代码和路径）
codegraph telemetry off

它几乎零配置。20 多种语言（TS/JS/Python/Go/Rust/Java/C#/Swift/Kotlin… 一直到 Vue/Svelte/Astro）靠扩展名自动识别，不用你一个个配。node_modules、dist、.venv 这些目录、还有 .gitignore 里的东西、大于 1MB 的文件，默认全部跳过——图里只有你自己的代码，没有第三方噪音。

真要调，也就一个可选的 codegraph.json，比如把某个提交进仓库的 vendored 主题排除掉，或者给非标准扩展名指个语言：

{
  "exclude": ["static/", "**/vendor/**"],
  "extensions": { ".tpl": "php" }
}

如果你想把它嵌进自己的程序里（比如某个 Electron 应用的主进程），npm 包还直接导出了 CodeGraph 类，import 和 require 都能拿到，自己 init、searchNodes、buildContext 一条龙；只是这条路跑在你自己的运行时上，需要 Node 22.5+（用的是内置的 node:sqlite）。CLI 和 MCP 那套不受影响，它们跑在自带的 runtime 上。

还有一点对我这种谨慎的人很重要：全程本地。源码和那个 SQLite 库都不离开你的机器，没有 API key、不连外部服务。给公司私有仓库建图，心里踏实。

完整命令和配置可以翻官方文档站，更新很勤，写得也清楚。

用之前，有几个点值得注意

最容易踩的一个：CodeGraph 只在 agent 直接查询它的时候才省事。如果你的 agent 习惯把”先去仓库探一探”这件事外包给一个专门读文件的子 agent（比如 Explore 类 sub-agent），那子 agent 照样一个个 read 文件，CodeGraph 反而成了白挂的一层。官方为此特地在给 agent 的指令里写明”自己直接答，别把探索甩给读文件的子 agent”——用 codegraph install 接 Claude Code 时这段会自动写进去，但你要是手动接、或者套了自定义工作流，记得让主 agent 自己查。

第二个点跟自动同步有关。前面说”索引永不过期”有个前提：那个文件监听的守护进程得能跑起来。要是你在沙箱环境里、或者显式设了 CODEGRAPH_NO_DAEMON=1 把它关了，watcher 就不工作了，这时候改完代码得自己手动同步一下：

# 守护进程没跑时（沙箱 / CI / 受限容器），手动增量同步
codegraph sync

本地正常用基本碰不到，但在 CI 或者受限容器里跑要心里有数。

最后一点是心态问题：这工具还很年轻、迭代很快（写这篇时最新版才发布没多久），命令和默认行为往后都可能再变。要是你想要个稳定的版本，别一直跟最新，锁一个就好：

# 看看有没有新版
codegraph upgrade --check

# 锁定到指定版本，不自动跟最新
codegraph upgrade 1.1.2

如果你天天用 Claude Code 或者 Cursor 啃大项目，CodeGraph 这种”先建图、再让 agent 查图”的路子，比起一遍遍 grep，体感差距是实打实的。

下一步很简单，挑一个你最熟的项目，跑这一条命令开始：

codegraph init

觉得有用就点个收藏吧——回头接自己项目的时候，照着这三步走一遍就行，省得再翻。
