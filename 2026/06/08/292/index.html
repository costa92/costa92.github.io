把 Claude Code 接到 Prometheus + Grafana：14 个变量讲清楚

我前两天帮一个团队 debug 他们的 Claude Code telemetry，把 alias 拷过去之后 Prometheus 一条数据都没收到。一查发现：默认 protocol 是 gRPC 不是 http/protobuf，端口是 4317 不是 4318，而对方的 Prometheus 只开了 HTTP receiver。一个变量错位，整条数据流哑火。

Claude Code 的 OpenTelemetry 配置一共有 14 个常用环境变量，分四层：开关、传输、隐私、基数。每一层都有”默认值跟你想要的不一样”的坑。

这篇分两半。前半把 14 个变量按层讲清楚（含完整对照表）。后半给你一份贴上就能跑的 Prometheus 2.47+ 原生 OTLP receive + Grafana 的 docker-compose——不需要 OTel Collector 中转，最少零件。

14 个 env vars 分四层看

按功能分组，记住每层只解决一个问题，记起来就简单：

层变量默认值决定

CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFICunset关 Anthropic 自己的遥测（与你的 OTel 无关）
OTEL_METRICS_EXPORTERunsetotlp / prometheus / console / none
OTEL_LOGS_EXPORTERunsetotlp / console / none（注意没有 prometheus）
传输OTEL_EXPORTER_OTLP_PROTOCOLgrpc ⚠️grpc / http/protobuf / http/json

OTEL_EXPORTER_OTLP_HEADERSunsetAuthorization=Bearer xxx

标⚠️的两个最坑：默认 protocol 是 gRPC，但很多公司 HTTPS 反代只允许 HTTP；默认带 session.id，每次 Claude 重启都是新 series，几天 Prometheus 内存就崩。

最小启用模板（本地 dev，推到本地 4317）：

export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_METRICS_EXPORTER=otlp
export OTEL_LOGS_EXPORTER=otlp
# 默认就是 grpc + localhost:4317，可以不写

跑一下 claude --version 之后用 tcpdump -i lo port 4317 抓包，如果有流量，说明开关层 OK。

debug 阶段不想搭后端时，最方便的是 console exporter——直接把 metric 打到 stderr：

export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_METRICS_EXPORTER=console
export OTEL_METRIC_EXPORT_INTERVAL=5000   # 5 秒一次，看得见
claude
# 然后看 stderr，会有 JSON 形式的 metric 流出来

三种 exporter：gRPC / HTTP/protobuf / 内置 Prometheus

我推荐先想清楚走哪条路再配，不然变量改来改去会乱：

Path A：gRPC OTLP（默认，最快）

export OTEL_EXPORTER_OTLP_PROTOCOL=grpc
export OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317

适合本地 / VPC 内部、能直连的场景。性能最好，但很多企业反代不放 gRPC。

Path B：HTTP/protobuf OTLP（跨网络、跨反代最稳）

export OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
export OTEL_EXPORTER_OTLP_ENDPOINT=https://usage.example.com/api/v1/otlp
# Prometheus 2.47+ 直接收，无需 OTel Collector

这是 HelloTalk 那种远程聚合常用的姿势——HTTP/2 走 443，反代不挑食。我自己生产也用这条。

Path C：内置 Prometheus exporter（最简单）

export OTEL_METRICS_EXPORTER=prometheus
# Claude Code 会在本地起一个 /metrics 端点，Prometheus 主动 scrape

零依赖，本地用最快。缺点：只能采 metrics，logs 没法走这条；多用户聚合也不方便。

我自己的选型经验：单机本地用 Path C，团队聚合用 Path B，重度场景（要做 trace 关联）才上 Path A + OTel Collector。

传输进阶：per-signal 拆分 + 4 种认证姿势

走到生产环境，基本会遇到两个需求：metrics 和 logs 想推到不同后端、跨网络要带认证。Claude Code 都支持，只是参数容易写错。

Metrics 和 logs 走不同后端——比如 metrics 进 Prometheus、logs 进 Loki：

# 通用 endpoint 不写，用 per-signal override
export OTEL_EXPORTER_OTLP_METRICS_PROTOCOL=http/protobuf
export OTEL_EXPORTER_OTLP_METRICS_ENDPOINT=http://prometheus:9090/api/v1/otlp
export OTEL_EXPORTER_OTLP_LOGS_PROTOCOL=grpc
export OTEL_EXPORTER_OTLP_LOGS_ENDPOINT=http://loki:4317

注意 metrics endpoint 走 HTTP（带 /api/v1/otlp 路径），logs 走 gRPC（只到端口）。两边都各自的协议参数，不会互相覆盖。

⚠️ Claude 2.1 实测坑：单独设 OTEL_EXPORTER_OTLP_METRICS_ENDPOINT + OTEL_EXPORTER_OTLP_LOGS_ENDPOINT，实际只有 metrics endpoint 生效，logs 信号仍走通用 endpoint。我自己的环境（2.1.153）跑下来就是这个行为——Prom 拿到了、loki 静默落空。最稳的做法是只设 OTEL_EXPORTER_OTLP_ENDPOINT 指向一个 OTel Collector,让 Collector 内部按 pipeline 分流给 Prom 和 Loki。Collector 这层配置 12 行就够，下面”Docker 容器”章节会贴完整 yaml。

认证姿势 1：静态 Bearer token——最简单，token 写在环境变量里：

export OTEL_EXPORTER_OTLP_HEADERS="Authorization=Bearer $(cat /etc/secrets/otel-token)"
# 多个 header 用逗号分隔
export OTEL_EXPORTER_OTLP_HEADERS="Authorization=Bearer xxx,X-Tenant-ID=team-platform"

认证姿势 2：mTLS——企业内 zero-trust 网络常用，OTLP exporter 直接吃 cert 文件路径：

# 客户端证书三件套（与 OTLP_PROTOCOL 配合）
export OTEL_EXPORTER_OTLP_CLIENT_CERTIFICATE=/etc/ssl/claude-client.crt
export OTEL_EXPORTER_OTLP_CLIENT_KEY=/etc/ssl/claude-client.key
export OTEL_EXPORTER_OTLP_CERTIFICATE=/etc/ssl/ca-bundle.crt   # 服务端 CA

认证姿势 3：动态 headers——token 会过期、要轮换，写个脚本输出 JSON：

# 准备一个 helper 脚本，stdout 输出 JSON
cat > /usr/local/bin/get-otel-headers.sh <<'EOF'
#!/bin/bash
TOKEN=$(aws sts assume-role-with-web-identity ... | jq -r '.Credentials.SessionToken')
echo "{\"Authorization\": \"Bearer $TOKEN\"}"
EOF
chmod +x /usr/local/bin/get-otel-headers.sh

# 告诉 Claude Code 用这个脚本（仅 http/protobuf 和 http/json 支持，gRPC 不行）
export OTEL_EXPORTER_OTLP_HEADERS_HELPER=/usr/local/bin/get-otel-headers.sh
export CLAUDE_CODE_OTEL_HEADERS_HELPER_DEBOUNCE_MS=900000   # 15 分钟刷新一次

认证姿势 4：管理员配置文件——团队不想让每个开发者自己配 env vars，用 settings 文件集中管理：

// /etc/claude-code/managed-settings.json（管理员级，优先级最高）
{
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp",
    "OTEL_EXPORTER_OTLP_PROTOCOL": "http/protobuf",
    "OTEL_EXPORTER_OTLP_ENDPOINT": "https://usage.example.com/api/v1/otlp",
    "OTEL_EXPORTER_OTLP_HEADERS": "Authorization=Bearer $(cat /etc/secrets/otel)",
    "OTEL_RESOURCE_ATTRIBUTES": "deployment.environment=prod"
  }
}

放好之后，开发者 claude 一启动就自动拿到这些 env vars，不需要自己 export，也不能覆盖。多团队场景下 OTEL_RESOURCE_ATTRIBUTES 可以叠加打标签：

# 用户 shell 里追加 team / user 维度（managed.json 里 prod 不变）
export OTEL_RESOURCE_ATTRIBUTES="${OTEL_RESOURCE_ATTRIBUTES},team=infra,user.email=$(git config user.email),cost_center=eng-123"

后端拿到的 resource attributes 是合并集，Prometheus 那边在 promote_resource_attributes 里挑要 promote 成 label 的字段就行。

隐私 + 基数：你不想全推的两类参数

这块是踩坑重灾区。默认值在 最保守 一端（什么都不上报），但很多人为了”看清楚一点”把开关全打开，结果一周后日志里塞满了源码片段被合规扫到。

隐私三件套（默认全是关，按需开）：

# 我个人推荐的"够用不越界"组合
export OTEL_LOG_USER_PROMPTS=0     # 用户 prompt 正文：永远关
export OTEL_LOG_TOOL_CONTENT=0     # tool 输入输出正文：永远关（会包含文件内容！）
export OTEL_LOG_TOOL_DETAILS=1     # tool 元信息：可以开，对审计有用

OTEL_LOG_TOOL_DETAILS=1 会上报 Bash 命令字符串、MCP server / tool 名、skill 名。这部分对审计很有用（知道用户在调哪些工具），但本身不含正文，比 TOOL_CONTENT 安全得多。

如果你要在 Grafana 的 prompt 二级页里看到 Tool Calls 的请求参数正文，还必须额外打开：

export OTEL_LOG_TOOL_CONTENT=1

注意这只对新启动的 Claude Code 进程生效，历史 Loki 数据补不回来。它会把 tool 输入/输出内容一起上报，可能包含文件路径、命令参数、代码片段甚至文件内容，只建议在明确的审计环境里打开。

还有一个高危隐私变量必须知道：OTEL_LOG_RAW_API_BODIES。它会上报完整的 Anthropic Messages API 请求 / 响应 JSON，包含整段对话历史和生成内容。开启它等于把所有上面三个开关一次性打开，并且体量翻倍——单次 prompt 体量可以到 60KB。除非你在做 LLM 审计平台，不然永远别开：

# 危险，除非你知道自己在做什么
# export OTEL_LOG_RAW_API_BODIES=1            # 内联，单条 60KB 截断
# export OTEL_LOG_RAW_API_BODIES=file:/var/log/claude-bodies   # 落盘，无截断

基数控制四件套（全部默认开的两个一定要关，另外两个按需）：

# 高基数自杀风险：必须关
export OTEL_METRICS_INCLUDE_SESSION_ID=false       # 默认 true，每次重启换值
export OTEL_METRICS_INCLUDE_ACCOUNT_UUID=false     # 默认 true，多用户场景按 team 聚合时不需要

# 默认关的两个：按需开
export OTEL_METRICS_INCLUDE_VERSION=true           # 默认 false。Claude Code 升级监控想看版本分布时打开
export OTEL_METRICS_INCLUDE_ENTRYPOINT=true        # 默认 false。区分 `claude` / `claude -p` / Agent SDK 调用源

session.id 每次 claude 命令重启都换，加进 metric 标签就是 cardinality 自杀。我之前没关，跑了 4 天 Prometheus 内存从 200MB 涨到 8GB，回查才发现是这个。VERSION 和 ENTRYPOINT 默认是关的——只在你要做”哪些用户还卡在旧版本””脚本调用占比多少”这类分析时才打开。

Prometheus 原生 OTLP 接收：一条 flag 搞定

Prometheus 从 2.47 开始原生支持 OTLP HTTP receive，不需要 OTel Collector 当中间件。这是目前最短的接收路径。

启动时加一个 flag：

prometheus \
  --config.file=/etc/prometheus/prometheus.yml \
  --web.enable-otlp-receiver

然后 prometheus.yml 里加一个 otlp: 段，告诉它把哪些 OTel resource attribute 提升成 label：

# prometheus.yml
otlp:
  # 把这些 OTel 资源属性 promote 成 Prometheus label，方便查询
  promote_resource_attributes:
    - service.name
    - service.instance.id
    - team
    - user.email
  keep_identifying_resource_attributes: true
  # translation_strategy 用默认（UnderscoreEscapingWithSuffixes）
  #   - OTel 名里 . → _（claude_code.session.count → claude_code_session_count）
  #   - counter 自动加 _total 后缀
  #   - 带单位的 metric 加单位后缀（详见下文"验证"段）

storage:
  tsdb:
    # OTLP 数据偶尔会乱序到达，给一点容忍
    out_of_order_time_window: 30m

Claude Code 端指过去：

export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_METRICS_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
export OTEL_EXPORTER_OTLP_ENDPOINT=http://prometheus:9090/api/v1/otlp
export OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=cumulative
export OTEL_METRICS_INCLUDE_SESSION_ID=false
claude

注意三个细节：endpoint 是 /api/v1/otlp（Prometheus 内部自动补 /v1/metrics）；temporality 一定要 cumulative，Prometheus 不擅长 delta；session_id 关掉。

验证：

curl -s http://localhost:9090/api/v1/query?query=claude_code_session_count_total | jq .

如果返回里有 result 数组，数据流就通了。

实测过的命名规则（默认 UnderscoreEscapingWithSuffixes 策略）：

OTel 原名Prometheus 名备注
claude_code.session.countclaude_code_session_count_totalunit=count 时不加单位后缀
claude_code.token.usageclaude_code_token_usage_**tokens**_total⚠️ unit=tokens,加单位
claude_code.cost.usageclaude_code_cost_usage_**USD**_total⚠️ unit=USD,加单位
claude_code.active_time.totalclaude_code_active_time_**seconds**_total⚠️ unit=s → seconds
claude_code.lines_of_code.countclaude_code_lines_of_code_count_totalunit=count
claude_code.code_edit_tool.decisionclaude_code_code_edit_tool_decision_totalunit=count

带单位的那 3 个最容易踩坑——按”原 OTel 名替换点”猜会查不到。我第一次跑就栽在 claude_code_cost_usage_total 上,debug 半天才发现要加 _USD。下面的 PromQL 都用了实测过的正确命名。

加 Grafana，跑一个 demo dashboard

Grafana 用 provisioning 模式自动加 data source + dashboard，比手点省心。

grafana/provisioning/datasources/prometheus.yml：

apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: false
    jsonData:
      httpMethod: POST
      timeInterval: 15s

几条我每天都会看的 PromQL，覆盖”成本 / 产出 / 谁在用”三个维度：

# 1. 各模型每小时 token 消耗（成本归因第一选）
sum by (model, type) (increase(claude_code_token_usage_tokens_total[1h]))

# 2. 各团队每天的 cost（USD）— 需要前面配了 team resource attribute
sum by (team) (increase(claude_code_cost_usage_USD_total[1d]))

# 3. 每天有多少独立 session 启动（adoption 跟进）
sum(increase(claude_code_session_count_total[1d]))

# 4. lines_of_code 增减（产出维度）
sum(rate(claude_code_lines_of_code_count_total{type="added"}[1h]))

# 5. 工具调用决策分布（被拒绝率，看权限模型健康度）
sum by (decision) (rate(claude_code_code_edit_tool_decision_total[5m]))

我自己 dashboard 上最常盯着的是第 2 条 + 第 5 条。前者管钱，后者看 Claude 是不是经常被人按 deny——按 deny 太多说明工具描述写得不好，或者用户对 Claude 不信任，都是要 fix 的信号。

⚠️ OTLP 累计 counter + increase() 的坑（实测）：Claude Code 走 OTLP 推的是累计 counter，而且只在值变化时才推一个样本（稀疏推送）。当某指标在你选的时间窗内只有 1 个样本时，increase(metric[$__range]) 因为”两点才能算增量”会返回空——dashboard 上就是一片 No data，空闲几分钟后打开尤其明显。两个稳妥替代：

  • 看”区间累计值”（如总成本、总 token、commit 数）用 sum(last_over_time(metric[$__range]))——取窗口内最后一个样本，顺便绕开 Prometheus 5 分钟 staleness 导致的 instant 查询变空；
  • 看”窗口增量趋势”，要么把范围拉到足够长让 increase() 有 ≥2 个样本，要么干脆把趋势计算挪到 Loki 端 sum_over_time({...} | unwrap <字段> [$__interval])——日志侧每条事件都是独立样本，不受稀疏 counter 影响，趋势更准。本文后面 Loki 章节的成本/token 趋势都用后一种。

如果你想直接拿一份现成 dashboard，Anthropic 自己开源了 anthropics/claude-code-monitoring-guide 仓库，里面有 docker-compose + dashboard JSON。我下面这份 docker-compose 是简化版（去掉了它的 Loki 部分，只留 Prometheus + Grafana）：

Dashboard 自动 provisioning + 告警规则

PromQL 写完，下一步是让 Grafana 启动时自动加载 dashboard 和 alert，开发机和生产环境出一份代码就够。

grafana/provisioning/dashboards/dashboards.yml：

apiVersion: 1
providers:
  - name: claude-code
    type: file
    disableDeletion: true
    updateIntervalSeconds: 30
    allowUiUpdates: false
    options:
      path: /etc/grafana/provisioning/dashboards/json
      foldersFromFilesStructure: true

然后把 dashboard JSON 放到 grafana/provisioning/dashboards/json/cost.json——下面是一个最小可用的成本归因面板骨架（一个 stat + 一个 timeseries + 两个变量 $team 和 $model）：

{
  "title": "Claude Code Cost Attribution",
  "templating": {
    "list": [
      {"name": "team", "type": "query", "datasource": "Prometheus",
       "query": "label_values(claude_code_cost_usage_USD_total, team)", "multi": true, "includeAll": true},
      {"name": "model", "type": "query", "datasource": "Prometheus",
       "query": "label_values(claude_code_cost_usage_USD_total, model)", "multi": true, "includeAll": true}
    ]
  },
  "panels": [
    {"id": 1, "type": "stat", "title": "Today's Total Cost (USD)",
     "targets": [{"expr": "sum(increase(claude_code_cost_usage_USD_total{team=~\"$team\",model=~\"$model\"}[1d]))"}]},
    {"id": 2, "type": "timeseries", "title": "Hourly Cost by Team",
     "targets": [{"expr": "sum by (team) (increase(claude_code_cost_usage_USD_total{model=~\"$model\"}[1h]))",
                  "legendFormat": "{{team}}"}]}
  ]
}

告警规则——团队成本超阈值时报警，写在 Prometheus 端（也可以用 Grafana 9+ 的 unified alerting）：

# prometheus-rules.yml，挂到 prometheus.yml 的 rule_files: 段
groups:
  - name: claude_code_cost
    interval: 1m
    rules:
      - alert: ClaudeCodeTeamCostOverBudget
        expr: |
          sum by (team) (increase(claude_code_cost_usage_USD_total[1d])) > 50
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Team {{ $labels.team }} 今日成本 ${{ $value | printf \"%.2f\" }} 超 $50 阈值"
      - alert: ClaudeCodeDenyRateHigh
        expr: |
          sum by (team) (rate(claude_code_code_edit_tool_decision_total{decision="deny"}[15m]))
            / sum by (team) (rate(claude_code_code_edit_tool_decision_total[15m])) > 0.3
        for: 10m
        labels:
          severity: info
        annotations:
          summary: "Team {{ $labels.team }} 工具拒绝率 > 30%，工具描述或权限模型可能要 fix"

第二条告警是我自己生产里很有价值的一条——deny 率高说明 Claude 推荐的命令经常被否决，要么是 prompt 写得不准，要么是工具描述不清楚，能直接反推 fix 方向。

Skill / Agent / MCP 使用分布：数据驱动砍/推哪些

装了 30 个 skill 装了 8 个 MCP server，半年后没人记得到底哪些在跑活，凭印象砍最容易砍错。Claude Code 的 OTel 数据其实把每次调用都打了标签——同一个 claude_code.token.usage metric 上额外带 skill.name / plugin.name / agent.name 维度，按它们 group by 就是一张”谁在烧 token”的天然 dashboard。

前提是 OTEL_LOG_TOOL_DETAILS=1 必须打开——不然下面所有 skill/plugin/agent 标签全是空字符串，dashboard 永远看不到：

export OTEL_LOG_TOOL_DETAILS=1
# 这会上报：Bash 命令字符串、MCP server/tool 名、skill 名、subagent 类型
# 不含：prompt 正文、tool 输入输出 — 见前面"隐私 + 基数"章节的权衡

打开之后，Prometheus 端会看到 claude_code_token_usage_tokens_total 上多出三个标签：skill_name / plugin_name / agent_name（OTel 名里的 . 在 Prom 端转成 _）。

⚠️ 重要实测发现：skill_name 和 plugin_name 标签的值不是 skill 的具体名(如 using-superpowers)，而是分类('third-party' / 推测还有 'built-in' 等)。Claude Code 应该不想把私有 skill 名上报到 metric。所以下面这条 topk(10, sum by skill_name) 拿到的是”分类型的 token 分布”(third-party vs built-in)而不是”哪 10 个 skill 最烧 token”——别被 PromQL 表达式骗了。agent_name 倒是具体的（如 general-purpose、自定义 subagent 名）。想要具体 skill 名（全名）别在 Prometheus 这边死磕——走后面 Loki 章节的 skill_activated 事件，那里 skill_name 是真名。

按 skill 分类看 token 分布——分清楚是 third-party 插件还是内置 skill 在烧钱：

# 实测值通常只有 1-2 类:third-party / built-in
sum by (skill_name) (claude_code_token_usage_tokens_total{skill_name!=""})

# 如果只想看本周新增
sum by (skill_name) (increase(claude_code_token_usage_tokens_total{skill_name!=""}[7d]))

每个 plugin 分类的成本归因（USD）——类似上,跟 skill_name 通常一致:

sum by (plugin_name) (claude_code_cost_usage_USD_total{plugin_name!=""})

# 团队 × plugin 分类的二维视图
sum by (team, plugin_name) (claude_code_cost_usage_USD_total{plugin_name!=""})

Subagent 调用密度——agent_name 是真正的具体名（general-purpose / code-reviewer / 自定义 subagent 名），这条最有用：

# Top 10 烧 token 的 subagent
topk(10,
  sum by (agent_name) (claude_code_token_usage_tokens_total{agent_name!=""})
)

# 配合 user_email 看是哪个开发者疯狂调 subagent
topk(5,
  sum by (user_email, agent_name) (rate(claude_code_token_usage_tokens_total{agent_name!=""}[1h]))
)

要看具体 skill / MCP server / 工具名,走 logs 后端(Loki/Elastic)最直接。这里有一组第一版踩过坑、必须讲清楚的事实：Claude Code 走 OTel 上报时，所有 attribute 经 OTel Collector 写成 Loki 3.x 的 structured metadata（既不是 stream label，也不是 JSON body），并且属性名里的 . 全部转成 _——skill.name→skill_name、process.pid→process_pid、event.name→event_name、server.name→server_name。LogQL 查询一律用下划线名。

🔑 structured metadata 怎么过滤（关键，纠正一个广为流传的误解）：

  • ❌ 不能放进 stream selector 大括号——{service_name="claude-code", tool_name="Skill"} 直接返 0（大括号只认真正的 stream label，本环境只有 service_name 一个）；
  • ✅ 能用 pipeline label filter（管道符）{service_name="claude-code"} | event_name="tool_result" | tool_name="Skill" 精确过滤，实测稳定返回数据，也能 sum by(tool_name) 聚合、| unwrap duration_ms 取数值。

一句话：过滤 structured metadata 用管道符 | key="val"，不是大括号 {key="val"}。记牢这条，下面所有按 skill / agent / server 精确过滤的查询都能跑通——不必退回 |= 子串匹配那种容易误伤的写法。

按 event_name 区分事件类型后，最有价值的几类：

  • skill_activated — 每次 skill 激活打一条，直接带具体 skill 名：skill_name（superpowers:using-superpowers / feishu-docs 这种全名）、skill_source（plugin / userSettings / project）、invocation_trigger（user-slash / nested-skill / claude-proactive）。这是拿”具体哪个 skill 在跑”的正解——不用像旧版那样绕 tool_result 按行人工翻（详见下面「找具体 skill 名」小节）。
  • subagent_completed — 每个子代理跑完打一条，带 per-run 明细：agent_type（general-purpose / Explore / 自定义名）、total_tokens、total_tool_uses、duration_ms、model、agent_source（built-in / 自定义）。比 Prometheus 的 agent_name 聚合多了”每一次跑了多久、用了多少工具/token”的粒度。
  • mcp_server_connection — 每次 MCP server 连接握手打一条。server_name 是具体名（context7 / sequential-thinking / n8n-mcp 等）；status 是连接结果（connected / failed / disconnected）——这是 MCP 健康度的关键信号（实测 failed 常常 ≥ connected，多发生在 sse/远程 server）；还有 transport_type（stdio 本地 / sse·claudeai-proxy 远程）、server_scope、duration_ms（握手耗时）。
  • tool_result — 每次工具调用完打一条。注意（2.1.x 实际行为）：tool_name 对 MCP 工具始终是字符串 mcp_tool（不区分 server），真实 server+tool 名在另一个 structured metadata 字段 tool_parameters 里，是 JSON 字符串，例如 {"mcp_server_name":"context7","mcp_tool_name":"resolve-library-id"}。Loki 的 | json 只能解 log body、解不了 structured metadata 里的 JSON value，要拆 server 出来得在客户端（Python / jq）解，或用 line_format 把整串打出来人工看。

📌 更正一则旧版误解:本文早期版本写过”Loki 3.3 下用精确 match 偶尔返空、得退回 line filter |= 子串匹配”——复盘下来，那是把 stream selector {key="val"}（structured metadata 放这里确实返 0）和 pipeline filter | key="val"（完全可行）搞混了。实测 | event_name="mcp_server_connection"、| tool_name="Skill"、| status="failed" 这类管道符过滤都稳定生效。下面的例子统一用管道符精确过滤；|= 子串匹配只在”搜某个关键词出现在哪”时才用，不再拿来当 label 过滤的替代。

列出本会话连接过的所有 MCP server(实测 12 个:context7 / chrome-devtools / mixsearch / sequential-thinking-1 / zread / web-search-prime / web-reader / vibe_kanban / playwright / n8n-mcp / plugin:superpowers-chrome:chrome / sequential-thinking):

# 区间内连接过的 MCP server（管道符精确过滤 event_name）
sum by (server_name) (
  count_over_time({service_name="claude-code"} | event_name="mcp_server_connection" [$__range])
)

MCP 连接健康度（这是最该上 dashboard 的一张）——status 维度此前完全被”连接次数”掩盖，实测把它拆出来才发现 failed 往往比 connected 还多：

# 连接状态分布（connected / failed / disconnected）
sum by (status) (
  count_over_time({service_name="claude-code"} | event_name="mcp_server_connection" [$__range])
)

# 哪些 server 在掉线 —— 直接点名排障入口（实测集中在远程 sse server）
sum by (server_name) (
  count_over_time({service_name="claude-code"} | event_name="mcp_server_connection" | status="failed" | server_name!="" [$__range])
)

# 握手延迟 avg by server（unwrap 数值字段，慢连接拖慢启动）
avg by (server_name) (
  avg_over_time({service_name="claude-code"} | event_name="mcp_server_connection" | server_name!="" | unwrap duration_ms [$__range])
)

每次 MCP 工具调用的延迟、谁调的、参数(把 tool_parameters 这个 JSON label 解出 mcp_server_name 字段):

# 关键:Loki 3.3 的 | json 只能解 log body,解不了 structured metadata label 里的 JSON value
# (tool_parameters 是 OTel Collector 写到 Loki structured metadata 的 label)
# 所以 LogQL 只做 line filter 圈定范围,JSON 解在客户端做
{service_name="claude-code"} |= "tool_result"
  | line_format `{{.user_email}} → {{.tool_parameters}} | ok={{.success}} dur={{.duration_ms}}ms | in={{.tool_input_size_bytes}}B out={{.tool_result_size_bytes}}B`

实测一行: costa9293@gmail.com → {"mcp_server_name":"context7","mcp_tool_name":"resolve-library-id"} | ok=true dur=4763ms | in=43B out=1679B —— 哪个用户在哪个时刻调了哪个 MCP server 的哪个 tool、参数多大、返回多大,白纸黑字。要按 server 拆开看,客户端用 jq / Python 解 tool_parameters 这串 JSON 即可(Loki WebUI 有自带 JSON inspector)。

按 MCP server 分组看调用次数(1h,降序):

topk(20,
  sum by (server_name) (
    count_over_time({service_name="claude-code"} |= "mcp_server_connection" [1h])
  )
)

只看某个 server 的 tool_result(如 context7):

{service_name="claude-code"} |= "context7" |= "tool_result"

每次调用的输入参数(tool_input 也是 JSON label,审计时重要):LogQL 不解它,直接 |= "query":"<keyword>" 做 substring 搜索最实际:

# 找所有传了 query="fastapi" 的 MCP tool 调用
{service_name="claude-code"} |= "tool_result" |= "fastapi"

如果团队关心某个 MCP server 的健康度,直接在 Grafana 里把 server_name 设成变量,配上 rate(count_over_time(...[5m])) 就能看到连接成功率/失败率时间序列。这套组合实测在 Loki 3.3 + OTel Collector 下稳定工作。

找具体 skill 名：直接查 skill_activated（正解，别再绕 tool_result）

旧版这里花了很大篇幅教人”用 sum by(tool_name) 拿分布、再到 logs panel 按行人工找 Skill 那一行”——那是当时误以为 structured metadata 不能精确过滤、又没发现有专门事件记这件事。正解是：

# 用了哪些 skill、各多少次（具体全名，不是 third-party/built-in 分类）
sum by (skill_name) (
  count_over_time({service_name="claude-code"} | event_name="skill_activated" [$__range])
)

# skill 来源 / 触发方式分布
sum by (skill_source)       (count_over_time({service_name="claude-code"} | event_name="skill_activated" [$__range]))
sum by (invocation_trigger) (count_over_time({service_name="claude-code"} | event_name="skill_activated" [$__range]))

实测直接拿到 superpowers:using-superpowers / superpowers:systematic-debugging / feishu-docs 这种具体名，外加 skill_source=plugin/userSettings、invocation_trigger=user-slash/nested-skill/claude-proactive——比在 tool_result 里人工翻精确得多。

子代理 per-run 明细：查 subagent_completed

Prometheus 的 agent_name 只能给”某 agent 累计烧了多少 token”。要看”每一次跑了多久、用了多少工具”，查事件：

# 各子代理类型运行次数
sum by (agent_type) (
  count_over_time({service_name="claude-code"} | event_name="subagent_completed" [$__range])
)

# 每次运行的明细：类型 / 模型 / token / 工具数 / 耗时
{service_name="claude-code"} | event_name="subagent_completed"
  | line_format `{{.agent_type}} | {{.model}} | tok={{.total_tokens}} tools={{.total_tool_uses}} dur={{.duration_ms}}ms`

通用范式：structured metadata 一律用管道符精确过滤。下面这些旧版以为”Loki 返 0”的需求，换成管道符全部能跑：

想做的事❌ 旧版（返 0 / 容易误伤）✅ 正确写法
只看 Skill 工具调用{service_name="claude-code", tool_name="Skill"}{service_name="claude-code"} | event_name="tool_result" | tool_name="Skill"
只看某 server 连接|= "context7" 子串匹配| event_name="mcp_server_connection" | server_name="context7"
工具失败率（旧版做不了）| event_name="tool_result" | success="false" 占总数

旧版列的那张”❌ 多 label match 返 0 / ✅ 只能 sum by”的表，根因就是没区分 stream selector {} 和 pipeline filter \|。{key="val"} 对 structured metadata 确实返 0，但那不是 Loki 的限制，是写法放错了位置——挪到管道符后就好。

仍然有效的 tool_result logs panel（看每次调用的工具/参数/耗时，MCP 真实 server 在 tool_parameters JSON 里，客户端解）：

{service_name="claude-code"} | event_name="tool_result"
  | line_format `{{.tool_name}} | {{.tool_parameters}} | ok={{.success}} {{.duration_ms}}ms`

结合 duration_ms、tool_input_size_bytes / tool_result_size_bytes 能快速定位”哪些调用最重”。

所以”找出 Skill 调用”这种需求,在 Loki 3.3 + OTel Collector 这套组合下,唯一可靠的方式是 sum by (tool_name) 拿到分布,再用 logs panel 按行人工找 Skill 那一行。

也可以简化:只看 metrics 的话,最实用的是 agent_name——subagent 的具体名能直接 group by 出来,半年下来能看出”哪些自定义 agent 真用了,哪些是装着好看”。skill/plugin 那两个分类标签只能给你”第三方 vs 内置”的粗粒度判断,而且实测在 2.1.153 永远只有 third-party 一个值,dashboard 画出来是单色,信息量约等于 0。

我自己 dashboard 上常驻这三面板(都是确认有真实多样性的 label,不再用 skill/plugin 分类):

  • Tokens by Subagent — sum by (agent_name) (claude_code_token_usage_tokens_total{agent_name!=""})。实测 Explore 和 general-purpose 各 50K+ tokens,两条柱,信息密度高。
  • Cost by Model (USD) — sum by (model) (claude_code_cost_usage_USD_total)。不同模型价格差几倍,这条直接告诉你钱烧在哪类模型上。
  • Top 10 Users by Cost (1h) — topk(10, sum by (user_email) (claude_code_cost_usage_USD_total))。需要 OTEL_METRICS_INCLUDE_USER_EMAIL=true 才能让 user_email 标签有值(默认是 user_id hash)。这条帮我跟 manager 解释钱花在哪——数据胜过 vibe。

Prometheus 的 skill_name / plugin_name 标签确实只有”第三方 vs 内置”这种粗分类（实测常只有 third-party 一个值，画出来单色、信息量≈0），但想看具体 skill 名不必退回 tool_result 人工翻——走上面的 skill_activated 事件就好。MCP 真实 server/tool 仍在 tool_result 的 tool_parameters JSON 里（客户端解）。

成本归因再进一步：按 query_source 拆”交互 vs 后台自动”

api_request 事件带一个容易被忽略、却极有价值的字段 query_source——标明这次 LLM 调用是谁发起的。实测一拆才发现后台自动调用在悄悄烧钱：

# 成本按来源拆（Loki 端 unwrap cost_usd，精确、不受稀疏 counter 影响）
sum by (query_source) (
  sum_over_time({service_name="claude-code"} | event_name="api_request" | unwrap cost_usd [$__range])
)

实测分布：repl_main_thread（你手敲的交互）占大头，但 sdk、away_summary（你离开时 Claude 自动写的总结）、agent:*（子代理）、prompt_suggestion、generate_session_title 这些后台来源加起来能占三到四成——Prometheus 的 cost_usage 按 model/team 拆完全看不到这一层。想压成本，先看这张图里哪些后台来源能关。

把 Claude Code 跑在 Docker 容器里

很多人 dev 环境是 host 直跑 Claude Code，但越来越多的团队把它装进容器——一是隔离权限避免一不小心 rm -rf 到宿主机，二是多个项目环境互不污染，三是 OTel 配置可以跟容器走，不污染开发者本地 shell。

第一步：把 OTel 配置抽到 .env 文件——容器化的第一收益就是配置和镜像分离：

# otel.env（提交到 git 里也安全，没有 secret）
CLAUDE_CODE_ENABLE_TELEMETRY=1
OTEL_METRICS_EXPORTER=otlp
OTEL_LOGS_EXPORTER=otlp
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_EXPORTER_OTLP_ENDPOINT=http://prometheus:9090/api/v1/otlp
OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=cumulative
OTEL_METRICS_INCLUDE_SESSION_ID=false
OTEL_METRICS_INCLUDE_ACCOUNT_UUID=false
OTEL_LOG_TOOL_DETAILS=1
OTEL_RESOURCE_ATTRIBUTES=team=infra,deployment.environment=dev

Secret 单独放，永远别和 .env 混在一起：

# otel.secret.env（gitignore 加上！）
OTEL_EXPORTER_OTLP_HEADERS=Authorization=Bearer xxxxxxxxxxxxx

第二步：docker run 一次性起一个 Claude Code 容器——适合临时调一下：

docker run -it --rm \
  --name claude-dev \
  --network monitoring \
  --env-file ./otel.env \
  --env-file ./otel.secret.env \
  -v "$HOME/.claude:/root/.claude" \
  -v "$(pwd):/workspace" \
  -w /workspace \
  node:24-bookworm \
  bash -lc "npm i -g @anthropic-ai/claude-code >/dev/null && claude"

三个 -v 挂载点要看清：~/.claude 装登录态和 settings，挂进去之后容器里的 claude 一启动就是登录状态；$(pwd) 是当前项目工作区；--network monitoring 是为了能解析到前面 docker-compose 起的 prometheus 容器名。

第三步：把 Claude Code 加进 docker-compose——适合常驻 dev container 模式：

# 加到前面 docker-compose.yml 的 services: 段下
  claude:
    image: node:24-bookworm
    container_name: claude-dev
    networks: [default]
    env_file:
      - otel.env
      - otel.secret.env
    volumes:
      - "$HOME/.claude:/root/.claude"
      - "$PWD:/workspace"
    working_dir: /workspace
    tty: true
    stdin_open: true
    command: ["bash", "-lc", "npm i -g @anthropic-ai/claude-code && sleep infinity"]

挂上去之后 docker compose up -d claude，再 docker compose exec claude claude 就进入交互式 Claude，OTel 数据自动推到同 compose 里的 Prometheus 容器。

Secret 管理：上面用 --env-file otel.secret.env 是最简单的姿势，但 secret 文件还是落盘的。要更严格可以用 Docker secrets（compose 里 secrets: 段）或者在容器里跑一个 OTEL_EXPORTER_OTLP_HEADERS_HELPER 脚本去调云厂商的 KMS 拿临时 token——参考前面”传输进阶”章节那条动态 headers 的写法，把脚本 COPY 进容器即可。

完整 docker-compose：贴上就能跑

把这份 docker-compose.yml + 两个配置文件放一个目录，docker compose up 就能起：

# docker-compose.yml
services:
  prometheus:
    image: prom/prometheus:v3.5.0
    command:
      - --config.file=/etc/prometheus/prometheus.yml
      - --web.enable-otlp-receiver
      - --storage.tsdb.retention.time=30d
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - prom_data:/prometheus

  grafana:
    image: grafana/grafana:12.2.0
    ports:
      - "3000:3000"
    environment:
      GF_SECURITY_ADMIN_PASSWORD: admin
      GF_AUTH_ANONYMOUS_ENABLED: "true"
      GF_AUTH_ANONYMOUS_ORG_ROLE: Viewer
    volumes:
      - ./grafana/provisioning:/etc/grafana/provisioning:ro
      - grafana_data:/var/lib/grafana
    depends_on:
      - prometheus

volumes:
  prom_data:
  grafana_data:

prometheus.yml 和前一节那份一致，grafana/provisioning/datasources/prometheus.yml 也是前一节那份。目录长这样：

.
├── docker-compose.yml
├── prometheus.yml
└── grafana
    └── provisioning
        └── datasources
            └── prometheus.yml

起服务 + 在你常用 Claude Code 的 shell 配 env vars 指过去：

docker compose up -d
export CLAUDE_CODE_ENABLE_TELEMETRY=1
export OTEL_METRICS_EXPORTER=otlp
export OTEL_LOGS_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:9090/api/v1/otlp
export OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=cumulative
export OTEL_METRICS_INCLUDE_SESSION_ID=false
export OTEL_RESOURCE_ATTRIBUTES="team=infra,user.email=$(git config user.email)"

claude  # 用一会儿

5 分钟后打开浏览器访问 localhost:3000（Grafana），data source 已经自动连好 Prometheus，在 Explore 里输入 claude_code_session_count_total 应该能看到至少 1 个 series。

启用 telemetry 容易，把数据通起来难。把 14 个变量按”开关 / 传输 / 隐私 / 基数”四层记，知道每层只解决一个问题，配错了也容易定位是哪一层。

最大的两个默认值坑：protocol 默认是 gRPC（不是 http/protobuf）、session.id 默认带（基数会爆）。剩下的都是常识。

先收藏再看，里面的 docker-compose 是真能直接 docker compose up、跑得起来的最小集——下次起 OTel 接收端就别再现搜配置了。
