循环工程：从写提示词，到设计让 Agent 自己跑的循环

Anthropic 负责 Claude Code 的 Boris Cherny 说过一句话：「我不再给 Claude 写提示词了。我让一堆循环替我写、替我判断该干什么。」造出最常用编码 Agent 的人都不再手动敲提示词，这事值得认真看。

一句话讲清楚：循环工程（Loop Engineering）就是把你自己从「按提示词的人」，换成「设计那套按提示词的系统的人」。 Addy Osmani 在 2026 年 6 月 7 日的这篇博客里把这个说法系统化了——你写一小段程序，可能是个 shell 循环，可能是个定时任务，由它去跟模型对话，而不是你本人。

Boris Cherny 的完整说法是：「我不再给 Claude 写提示词了。我有一堆循环跑着，替我给 Claude 写提示词、替我想清楚该干什么。」当造出最常用编码 Agent 的那批人都说自己已经停手不再手动敲了，这件事就从小圈子的玩法变成了值得正经对待的范式。

下面拆三件事：这套范式到底变了什么、一个循环由哪些零件拼成、以及它有哪些你绕不开的代价。

提示词工程为什么走到了头

过去两年，从编码 Agent 里拿东西的方式很固定：写个好提示词，喂够上下文，读它的回复，再写下一句。你打一行，看返回，再打下一行。Addy 的判断是，这部分基本结束了。

你:   实现用户登录接口，用 JWT
Agent: [给出代码]
你:   加上 refresh token
Agent: [改]
你:   测试没过，错误码 401
Agent: [再改]
... 你全程在场，一句一句喂

问题不是 Agent 不够强，而是你成了瓶颈——它每迭代一步都要停下来等你拍板。模型一年比一年能扛长任务，可你的注意力没变多。当一个任务要二十轮来回，你就被钉在椅子上当那根串行的导线：复述约定、贴上下文、确认下一步。我自己最烦的就是每开一个新会话，又要把项目的目录结构、命名习惯、测试怎么跑重新讲一遍。

Peter Steinberger 把替代方案讲得很直接：你不该再给编码 Agent 写提示词了，你该设计那些给 Agent 写提示词的循环。这句话的重量在于，说它的人正是天天用这些工具的人，而不是旁观者。

# 最朴素的循环：把"判断该干什么"也交出去
# pick_next_task 是你自己写的挑活函数（从 issue / CI / backlog 里取一件）
while true; do
  task=$(pick_next_task)
  claude -p "完成这件事：$task"      # 让 Agent 干
  sleep 300
done

这段循环里，你已经不在每一步出现了。Claude Code 干脆把这件事做成了内建命令。

/loop 30m 检查 CI 失败并尝试修复    # 每 30 分钟跑一轮
/goal 把测试覆盖率提到 80%           # 给一个目标，自己迭代到达成

两个命令的差别值得停一下：/loop 给的是节奏（每隔多久跑一次），/goal 给的是终点（达成某个条件才停）。前者适合”持续盯着某类问题”，后者适合”一件有明确完成标准的事”。

Addy 把循环定义成「一个递归的目标：你定义一个目的，AI 反复迭代直到完成」。活没变轻松，变的是杠杆点的位置——从”敲提示词”挪到了”设计系统”。提示词工程没有消失，它被降级成了循环内部的一个零件：你仍然要写好那段触发提示词，只是不再亲手把它喂进去。

循环的骨架：自动化、worktree 和 skill

一个像样的循环，Addy 拆成五个构件外加一层记忆。先看撑起骨架的三个，加上那层缺不得的记忆。

自动化是循环的心跳。 它在固定节奏上做发现和分诊——定时触发、git 事件触发、或者 CI 信号触发。Addy 强调过：正是这个会自己醒来的触发器，才让一个循环真的成其为循环，而不是一次手动运行。没有心跳，你写得再漂亮的 prompt 也只是个一次性脚本。

# 自动化 = 循环的心跳：定时触发发现和分诊
# crontab：每天早上 9 点跑一次
0 9 * * * cd /repo && claude -p "$(cat triage-prompt.md)" >> loop.log

worktree 解决并行打架。 同时跑多个 Agent 时，最怕它们改同一批文件互相覆盖。git worktree 给每个任务一份隔离的工作目录。

# 每个并行任务一个独立 worktree，互不踩踏
git worktree add ../fix-issue-142 -b fix/issue-142
git worktree add ../fix-issue-153 -b fix/issue-153
git worktree list

skill 把项目知识固化下来。 Addy 有个说法叫”意图债”——你每次开新会话重新解释一遍约定，就是在反复还这笔债。skill 是一次性还清的方式：一个 SKILL.md 就是一段会被反复调用的说明书，外加可选的脚本和参考文件，让循环每次跑都不必重新教一遍。

---
name: triage
description: 读取昨天的 CI 失败和新开 issue，分类并写入待办
---
# 分诊流程
1. 拉取过去 24h 的 CI 失败记录
2. 按「能自动修 / 需人介入」分两类
3. 把结论写进 backlog.md

还有一层不算构件、却缺不得的东西：外部记忆。对话窗口会断、会被压缩，进度不能跟着断——循环把状态写进一个 markdown 文件或 Linear 看板，下一轮醒来先读它，知道昨天干到哪、哪些还没碰。这层记忆活在对话之外，所以换模型、清上下文都不影响它。

# loop-state.md（循环的记忆）
### 进行中
- [ ] #142 修复登录 401（worktree: fix-issue-142）
### 已完成
- [x] #138 补全 README 安装步骤

让循环能动手、会自查：连接器和子 Agent

剩下两个构件最关键——一个让循环碰得到真实世界，一个让它别自己批改自己的作业。

只能读文件系统的循环，只能提建议。 要让它真正动手——开 PR、更新工单、发 Slack、查数据库——靠的是 MCP（Model Context Protocol）连接器。按 Anthropic 的定义，MCP 是「连接 AI Agent 与外部系统的开放标准」。

{
  "mcpServers": {
    "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] },
    "linear": { "command": "npx", "args": ["-y", "mcp-linear"] }
  }
}

MCP 自 2024 年 11 月发布以来铺得很快，按 Anthropic 的说法，社区已经造了上千个 MCP 服务器，主流语言都有 SDK，行业把它当成了事实标准。一个 Agent 实现一次 MCP，就接上了一整个工具生态。

Anthropic 给的一个数字很能说明问题：用代码方式调用 MCP 工具，能把一次任务的 token 从 15 万压到 2000，省掉 98.7%（见 code execution with MCP）。这对长期跑着的循环不是小数——省下的是真金白银。Codex 和 Claude Code 都讲 MCP，你为一个写的连接器，换到另一个上大多直接能用，不用为换工具重写一遍集成。

# claude mcp 只管配置：list 看已接入的 server
claude mcp list
# github / linear ... 具体工具（open_pr、create_issue）由模型在会话内调用

子 Agent 把”写”和”查”分开。 这是循环里最有用的一个结构性动作。Addy 的原话是：写代码的那个模型，给自己的作业打分时太手软了。让同一个 Agent 既写又审，它会本能地替自己辩护。第二个 Agent 换一套指令、有时换一个模型、调高推理档位，专门挑刺，就能抓住第一个说服自己糊弄过去的东西。

更细一点的分工是按角色拆：一个负责探索（读代码、搞清现状），一个负责动手（写改动），一个负责验证（只找问题）。每个角色给不同的指令，甚至不同的模型，谁也别替谁背书。

# .codex/agents/reviewer.toml —— 一个只负责挑刺的子 Agent
name = "reviewer"
model = "gpt-5-codex"
instructions = """
你是代码审查员。只找问题，不写新功能。
重点看：边界条件、错误处理、有没有测试。
"""

我自己把审查交给第二个 Agent 之后，第一个 Agent 那些”自我感觉良好”的改动才露馅——它会把一个没处理空值的函数写得理直气壮。社区已经有人把这类现成模式整理成了工具集（loop-engineering，含 loop-audit / loop-init / loop-cost）。

# 让审查子 Agent 过一遍刚写的改动
claude --agent reviewer -p "审查 fix/issue-142 分支的 diff"

一天的循环长什么样

五个零件单看都不稀奇，拼起来才有意思。Addy 给的样板是一个每天早上在仓库上跑的自动化，我把它顺着走一遍：闹钟一响，自动化醒来，它的 prompt 调一个分诊 skill，读昨天的 CI 失败、新开 issue、最近的 commit，把结论写进一个 markdown 文件或 Linear 看板。

心跳可以是定时器，也可以挂在 CI 事件上。用 GitHub Actions 当触发器，循环就跟着仓库的节奏走。

# .github/workflows/morning-loop.yml —— 用 CI 事件当心跳
on:
  schedule:
    - cron: "0 1 * * *"   # 每天 UTC 01:00
jobs:
  loop:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: claude -p "$(cat .loop/triage.md)"

这个状态文件就是循环的工作记忆。它读昨天的烂摊子，写下今天要处理的清单，下一轮自己再接上。整套流程里最值得玩味的，是它把”发现工作”也自动化了——你不再需要每天早上自己翻 issue 列表决定先修哪个。

分诊完，循环对着每件待办开隔离 worktree、让 fixer 子 Agent 起草修复、让 reviewer 子 Agent 对着项目 skill 审一遍，最后连接器开 PR、更新工单。每件事在自己的 worktree 里跑，互不干扰，跑挂了也只脏一个分支。

# 分诊 → 隔离 → 修 → 审 → 开 PR，全程无人值守
# read_triaged 是你自己的解析函数，从 backlog.md 读出已分诊的 issue 号
for issue in $(read_triaged backlog.md); do
  git worktree add "../wt-$issue" -b "fix/$issue"
  ( cd "../wt-$issue" \
      && claude --agent fixer    -p "修复 $issue" \
      && claude --agent reviewer -p "审查这次改动" \
      && gh pr create --fill --label "loop-generated" --draft )  # 连接器/gh 开 PR
done

跑完一轮，状态文件被改写成次日早上你能一眼看懂的样子——哪些进了 PR、哪些被标记成需要人介入。

# loop-state.md（次日早上自动更新）
### 今早处理
- [x] #142 → PR #210 已开，待人审
- [ ] #155 改动过大，已标记需人介入

你早上真正要做的，是看循环昨晚干了什么、决定哪些值得合并。这一步永远在你手上。

# 你的活：看循环昨晚干了什么，决定哪些合并
gh pr list --label loop-generated
git log --oneline fix/issue-142

第一个循环该怎么起步

看完五个构件，别急着上 GitHub Actions 全自动。循环最稳的起法是先做小、先做得能随时叫停，让它在你看得见的地方把行为暴露出来。

# 第一步：只让它分诊和提建议，先别动代码
claude -p "读 backlog.md，列出今天该修的 3 件事，但不要改任何文件"

先看它挑的活靠不靠谱，确认判断力没问题，再给它动手的权限。起步阶段我推荐用 /goal 而不是 /loop，因为前者有明确终点、错了有边界，后者一旦放久最容易在你没盯着的时候跑偏。下一步是把作用域钉死，别让它碰整个仓库。

# 用 label 把循环的作用域圈住，它只碰带这个标签的 issue
gh issue list --label "loop-safe" --json number,title

挑一类边界清楚的活先试——补测试、修 lint、更新文档，这些错了也好回滚。等行为稳定，再加一道人审闸门：循环只开 PR，合并永远留给你。

# 循环只开草稿 PR，是否合并由人决定
claude --agent fixer -p "修复 #142" && \
  gh pr create --label "loop-generated" --draft

draft 加人审，循环再勤快也越不过你这一关。手动跑顺几天、确认它不闯祸，才值得把它升级成定时循环。

# 跑稳几天后，才挂上定时器变成真正的循环
crontab -l | grep loop || \
  echo "0 9 * * * cd /repo && ./run-loop.sh" | crontab -

Addy 那套每天早上自动跑的循环，是终点而不是起点。一上来就全自动，等于把一个你还没摸清脾气的系统放出去替你改代码。

循环的三个局限，每个都得你扛

循环替你做不了的事，恰恰是最要紧的那几件。Addy 把它们说得很白。

验证还是你的活。 原话是：无人值守跑着的循环，也是无人值守犯错的循环。循环开的 PR 别盲合，至少过一道自动门。

# 循环开的 PR 别盲合，至少过一道自动门
gh pr checks 210                          # CI 全绿？
git diff main...fix/issue-142 | wc -l     # 改了多少行，心里有数

理解会欠债。 循环越快地交付你没写的代码，「存在的东西」和「你真正搞懂的东西」之间的缺口就越大。我自己的体会是：连着合并几个循环生成的 PR 后，再去 debug 一个线上问题，会发现有些模块根本没读过，得现学自己仓库里的代码。这笔债平时不响，出事那天利滚利。对付它没有巧办法，只能定期逼自己回读循环写的关键路径。

# 对循环生成的代码，强制自己回读关键路径
git log --author="loop" --since="1 week" --stat

判断会交出去，钱也会烧掉。 Addy 说，循环自己跑起来后，很容易就不再有自己的看法，它给什么你就收什么——这是比代码出错更隐蔽的塌方，因为你连”该不该这么做”都不再问了。再加上一句他反复强调的话：你绝对得小心 token 成本。一个无人盯着的循环，半夜可以替你把一个月的预算烧光，所以单轮要设上限，每天要回头看花了多少。

# 给循环装个成本刹车：每轮设美元上限，超了自动停
claude -p --max-budget-usd 5.00 "$(cat .loop/triage.md)"
# 会话里随时用 /usage 看本轮花了多少；月度封顶在 Claude Console 设 workspace spend limit

总结：把循环建起来，但人得留在工程师的位置上

Addy 的收尾很准——把循环建起来，但要像一个打算继续当工程师的人那样去建。循环把杠杆点从”敲提示词”挪到了”设计系统”，可验证、犯错、判断该不该合并这些事，没有一样能交出去。

循环适合边界清楚、能自动验证的活：修 CI、补测试、跑迁移、清 issue 队列。不适合需求还没想清、或者错一次代价很大的改动——那种你还是得一句一句盯着。想往深里读，Addy 的原文和 Anthropic 那篇用代码调用 MCP都值得看。

觉得有用，点个「在看」，让还在一句句敲提示词的同行也看到。
