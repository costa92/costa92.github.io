CodeGraph 进阶：装好之后，这 4 个用法才是真省事

📚 系列导航：本文是《CodeGraph 上手到进阶》第 2/2 篇。上一篇：CodeGraph 实战：给 AI agent 装上代码地图。

上一篇把 CodeGraph 装好、接上了 Claude Code，建完图它就自动生效了。但我观察下来，大多数人就停在这儿——以为”装好就完事了”，结果只用到了它最表层的能力。

其实装好只是起点。CodeGraph 真正省事的地方，是几个得你主动用起来的进阶玩法。这篇挑 4 个我自己天天在用、官方也明确支持、上一篇没展开的：怎么把 codegraph explore 问到点子上、改代码前怎么用 codegraph impact 和 codegraph affected 看波及范围、怎么把它当库嵌进自己的工具、以及怎么在一个会话里查好几个仓库。讲讲怎么把它榨干。

把问题问对，explore 才给你想要的

codegraph_explore 是它对外唯一的工具，但同样一个仓库，问得好不好，拿回来的东西差很远。官方在给 agent 的指令里其实写明了三类最适合它的问法。

第一类是”X 是怎么工作的”。别问”看下这个文件”，直接问机制：

# 问机制，它会把相关符号的源码 + 调用链一起给你
codegraph explore "登录请求从路由到写库整条调用链是怎么走的"

第二类是”X 怎么到达 Y”的流程追踪。这是它最能打的场景——因为它能跟住 grep 跟不动的动态分发（回调、接口到实现、框架路由）：

# 追一条跨越动态分发的流程
codegraph explore "一个 HTTP 请求是怎么最终触发 PaymentService 的"

还有个细节很多人不知道：你在问题里直接点名一个文件或符号，它就当 Read 用，返回那个文件当前带行号的源码。所以”读文件”这件事也能走 explore，不用切回内置工具：

# 点名文件 = 读它当前的源码（带行号，和 Read 一个形状）
codegraph explore "src/auth/session.ts"

反过来说，最浪费的问法是把它当搜索框用——“搜一下 login”这种，它能答，但你拿到的是一堆零散命中，跟 grep 没拉开差距。它的价值在”关系”而不是”匹配”：你问的是一条调用链、一个机制、一个符号的上下游，它才把预先建好的调用图用起来。

举个我自己遇到的例子：一个 React 项目里某个状态莫名其妙被刷新，grep 组件名根本追不到，因为中间隔着一层 context 的回调和重渲染。我前前后后 grep + read 折腾了快 20 分钟没结果，换成 codegraph explore "这个状态是被谁触发更新的"，它把回调那一跳也接上了，一次就指到真正的源头。这种动态分发的调用链，是它和纯文本搜索拉开差距的地方。

我的习惯是：凡是”这块代码怎么回事”的问题，先 explore 问一句，基本一次就够，省掉了 agent 一堆 grep+read 的探路。

改代码前，先看一眼会炸到谁

这是我觉得日常收益最高的一个用法。动一个函数之前，先问清楚它的波及范围，比改完才发现牵一发动全身强太多。

让 agent 改代码最怕的就是它”只看到眼前那个文件”——把一个被十几处调用的方法签名改了，自己以为改完了，结果一跑全是连锁报错。先用 impact 把影响半径摸清楚，agent 心里有数，要么一起改、要么提前提醒你这个动不得。我把”改公共方法前先 impact 一下”当成了固定动作。

codegraph impact 直接给你某个符号的影响半径，--depth 控制往外追几层：

# 改 UserService 之前，看两层以内会牵动什么
codegraph impact UserService --depth 2

# 也可以单独看谁调它、它调谁
codegraph callers handleRequest
codegraph callees processOrder

impact 给你的不是一个干巴巴的数字，而是顺着调用边一层层展开的受影响符号——谁直接调它、再往上谁调那些调用者。--depth 就是控制这个展开往外走几圈：改一个底层工具函数，往外两三层可能就牵动半个模块；改一个叶子方法，半径基本就它自己。看一眼这个范围，你心里对”这次改动是小手术还是大动干戈”就有数了。

把这个能力接进测试流程，价值更大。codegraph affected 顺着 import 依赖往上追，找出某次改动会影响哪些测试文件——改完只跑这部分，不用全量。配进 git hook 或 CI 是这样：

#!/usr/bin/env bash
# pre-push / CI：只跑被这次改动波及的测试
AFFECTED=$(git diff --name-only HEAD | codegraph affected --stdin --quiet)
if [ -n "$AFFECTED" ]; then
  npx vitest run $AFFECTED
fi

我在一个全量测试要跑 4 分多钟的项目里接了这个，改一两个文件时只跑被波及的那几个测试，CI 时间直接掉到 30 秒，比原来省了快 5 倍，等待体验完全不一样。它还支持自定义测试文件匹配，比如只挑 e2e：

codegraph affected src/auth.ts --filter "e2e/*"

impact 和 callers/callees 都能带 --json，所以你也可以把它接到自己的脚本里做判断——比如”改动碰到了核心支付模块就强制要求人工 review”，拿 JSON 出来解析一下就能写。这一点在团队流程里比单纯人看更靠得住。

不只在 agent 里：把 CodeGraph 当库嵌进自己的工具

很多人没注意到，CodeGraph 的 npm 包直接导出了编程 API，import 和 require 都能拿到那个 CodeGraph 类。也就是说你能把它嵌进自己的脚本或者程序里（比如某个 Electron 应用的主进程），自己驱动这张图。

什么时候会想这么干？最常见的是你想做一个 CodeGraph 之外的小工具——比如一个代码导航面板、一个”给我这个 bug 相关的所有代码”的内部命令、或者把它接到自己的 agent 工作流里当上下文来源。这些场景不走 MCP，直接调库更顺。其中 buildContext 我觉得最有意思：你丢一句自然语言任务进去，它帮你把相关的符号、代码、关系打包成一段现成的上下文（还能直接出 markdown），省得你自己拼。

一个最小例子，建图 + 搜符号 + 拿调用者 + 给一段任务组上下文：

import CodeGraph from '@colbymchenry/codegraph';

const cg = await CodeGraph.init('/path/to/project');
await cg.indexAll({
  onProgress: (p) => console.log(`${p.phase}: ${p.current}/${p.total}`),
});

const results = cg.searchNodes('UserService');
const callers = cg.getCallers(results[0].node.id);

// 给一段自然语言任务直接组装出相关代码上下文
const context = await cg.buildContext('fix login bug', {
  maxNodes: 20, includeCode: true, format: 'markdown',
});

想常驻盯着文件变化、用完收尾，也都有对应方法：

cg.watch();    // 自动同步：文件变了就增量更新
cg.unwatch();  // 停止监听
cg.close();    // 收尾

包里还顺带导出了一些更底层的积木，像 DatabaseConnection、QueryBuilder 这些，给那种想自己直接驱动图、不满足于高层方法的场景用；TypeScript 类型也一起带着，接起来不用自己补声明。

有一点要注意：这条路跑在你自己的运行时上，所以需要 Node 22.5+（它用了内置的 node:sqlite）。CLI 和 MCP 那套不受影响，它们跑在自带的 runtime 上，没这个版本要求。

一个会话，查好几个仓库

最后一个是 monorepo 和多仓库党会喜欢的。CodeGraph 的索引是按项目走的，每个项目自己 codegraph init 一次，建出各自的 .codegraph/。

monorepo 最烦的就是改 A 服务时想顺手看一眼 B 服务怎么调它，传统做法要么开两个窗口来回跳，要么 agent 把整个巨型仓库全量扫一遍——又慢又把无关代码塞进上下文。CodeGraph 的处理方式很对路：各服务各建各的图，互不干扰。

关键在于：查询时只要传一个 projectPath，就能在同一个 agent 会话里命中任意一个建过索引的项目——主服务、另一个子服务、甚至旁边那个仓库，来回切都行。

# 在某个项目目录里建好它自己的图
cd services/payment && codegraph init

# 各服务各自一份索引，互不干扰
cd ../user && codegraph init

碰到一个还没建索引的路径，它不会报错，而是回一句”这块你用内置工具吧”，干净地把决定权留给你——要不要为某个目录建图，始终是你说了算。对那种”只想给核心几个服务建图、不想全量扫”的大单体仓库，这个设计很省心。

我自己在一个前后端放一起的仓库里就是这么用的：后端服务建了图，前端那一大坨还在重构的代码暂时没建。问后端的调用链它走图、又快又准；偶尔碰前端，它自动退回 grep/read，不会因为”没索引”就卡住或者报错。这种”建到哪算到哪、没建也不添乱”的渐进式策略，比那种”要么全量索引要么没法用”的工具舒服多了——你可以从最该被理清的那块代码开始，慢慢铺。

装好只是把地图铺上，真正的省事是从你开始主动查这张图开始的。先挑你最熟的那个项目，下次改代码前先问它一句：

codegraph impact <你要改的那个函数名>

这几个用法值得收藏——下次接新项目或者动老代码时，照着试一遍，体感会很不一样。
